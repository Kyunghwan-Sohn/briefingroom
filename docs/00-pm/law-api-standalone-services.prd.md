# 법제처 Open API 독립 서비스 아이디어 분석

**작성일**: 2026-03-31
**분석 기준**: 법제처 API만으로 구현 가능한 독립 서비스 (보도자료 연동 없음)
**인프라 전제**: Python + SQLite + GitHub Pages + 텔레그램, API 일 100~1000회

---

## 현재 보유 자산 정리

| 자산 | 상태 | 활용 가능 수준 |
|------|------|--------------|

| korean-law-mcp 64개 도구 | 사용 가능 | 판례, 해석례, 헌재결정, 세무판정 등 |

| 텔레그램 채널 | 운영 중 | 봇 API 연동 완료 |

| GitHub Actions CI/CD | 무료 티어 | cron 기반 자동화 |

### 아이디어 1: 법령 변경 알리미 (LawWatch)

**한 줄 설명**: 내가 관심 있는 법령이 개정되면 즉시 텔레그램/이메일로 알려주는 구독형 알림 서비스

**타겟 사용자**:
- 기업 법무/컴플라이언스 담당자
- 변호사/법무사/세무사 등 전문직
- 규제 산업(금융, 의료, 식품) 종사자

**핵심 기능 3가지**:
1. **법령 워치리스트**: 사용자가 관심 법령을 등록하면 매일 개정 여부를 체크하여 변경 시 알림
2. **개정 내용 diff**: 이전 버전과 현재 버전의 조문 비교를 시각적으로 제공 (신구조문대비)
3. **키워드 알림**: 특정 키워드가 포함된 새 법령/개정 법령 발생 시 알림

**수익 모델**:
- 무료: 법령 3개 워치리스트, 일일 알림
- 프로: 월 9,900원 - 법령 30개, 실시간 알림, diff 제공
- 비즈니스: 월 49,000원 - 무제한, 팀 공유, API 접근, 주간 리포트

**기술적 구현 난이도**: 하
- 현재 law.py의 search_law() + 캐시 인프라 그대로 활용
- 일 1회 cron으로 워치리스트 법령 조회 -> 이전 캐시와 비교 -> 변경 시 텔레그램 전송
- API 호출: 워치리스트 법령 수만큼 (캐시로 대부분 절약)

**시장 규모 추정**:
- 국내 변호사 약 30,000명, 세무사 약 14,000명, 법무사 약 5,000명
- 기업 법무팀 약 3,000개 (상장사 + 중견기업)
- TAM: 전문직 + 기업 약 52,000명 x 월 9,900원 = 연 62억원
- SAM: 디지털 활용 전문직 20% = 연 12.4억원
- SOM: 초기 1% 침투 = 연 1,240만원 (Year 1 목표)

**경쟁사 대비 차별점**:
- 국가법령정보센터: 알림 기능 없음, 직접 검색해야 함
- 빅케이스/로앤비: 판례 중심, 법령 개정 알림은 부가 기능
- **차별점**: 법령 변경에 "특화된" 실시간 알림 + diff 뷰 + 텔레그램 즉시 전달

### 아이디어 2: 쉬운 법률 사전 (LawPlain)

**한 줄 설명**: 어려운 법조문을 AI가 쉬운 말로 풀어주고, 관련 판례/해석례까지 한 번에 보여주는 서비스

**타겟 사용자**:
- 법률 비전문가 일반 국민
- 창업자/자영업자 (사업 관련 규제 확인)
- 대학생/취준생 (공무원 시험, 법학 학습)

**핵심 기능 3가지**:
1. **AI 법률 번역**: 법조문 원문 + LLM이 생성한 쉬운 해설 + 실생활 예시를 나란히 표시
2. **관련 판례 연결**: 해당 조문과 관련된 대법원 판례를 자동 검색하여 "실제 적용 사례" 제공
3. **상황별 법률 검색**: "월세 보증금을 못 받았어요" 같은 일상 언어로 검색하면 관련 법조문 제시

**수익 모델**:
- 무료: 일 5건 조회, 기본 해설
- 프리미엄: 월 4,900원 - 무제한 조회, 판례 연결, PDF 다운로드
- 광고: 법률 상담 서비스/변호사 매칭 플랫폼 제휴 배너

**기술적 구현 난이도**: 중
- 법조문 조회: lawService.do 활용 (이미 get_article_text() 구현 완료)
- AI 해설: 자체 LLM으로 조문별 쉬운 설명 생성 + 캐시
- 판례 연결: precSearch.do로 해당 법령 관련 판례 검색
- 상황별 검색: LLM으로 사용자 질문 -> 법령 키워드 변환 -> lawSearch.do

**시장 규모 추정**:
- 법률 정보 검색 수요: 월 2,000만 건 (국가법령정보센터 기준)
- TAM: 법률 정보 검색 인구 약 500만명 x 월 4,900원 = 연 2,940억원
- SAM: 유료 의향 5% = 연 147억원
- SOM: 초기 0.1% = 연 1,470만원

**경쟁사 대비 차별점**:
- 국가법령정보센터: 원문만 제공, 해설 없음
- 네이버 지식iN/법률 상담: 답변 품질 불균일, 법조문 직접 연결 없음
- 로톡: 변호사 상담 중심 (건당 비용 높음)
- **차별점**: "조문 원문 + AI 해설 + 판례" 3단 구조를 무료/저가로 제공

### 아이디어 3: 업종별 규제 맵 (BizLaw)

**한 줄 설명**: 사업을 시작하거나 운영할 때 준수해야 할 모든 법령을 업종별로 정리해주는 규제 지도

**타겟 사용자**:
- 예비 창업자 (카페, 음식점, 쇼핑몰, 학원 등)
- 소상공인/자영업자
- 프랜차이즈 가맹점주
- 스타트업 대표

**핵심 기능 3가지**:
1. **업종 선택 -> 규제 체크리스트**: "카페 창업" 선택 시 식품위생법, 소방법, 건축법, 근로기준법 등 관련 법령 목록 + 핵심 조문 자동 제공
2. **규제 타임라인**: 사업 준비 -> 인허가 -> 운영 -> 폐업 단계별로 필요한 법적 요건 정리
3. **규제 변경 알림**: 해당 업종 관련 법령이 개정되면 자동 알림 (아이디어 1과 시너지)

**수익 모델**:
- 무료: 주요 10개 업종 기본 규제 맵
- 프로: 월 9,900원 - 전 업종 + 규제 변경 알림 + 체크리스트 PDF
- 기관 제휴: 소상공인진흥공단, 창업진흥원 등에 데이터 제공 (B2G)

**기술적 구현 난이도**: 중
- 업종-법령 매핑 DB 초기 구축이 핵심 (LLM으로 반자동 생성 가능)
- 법령 조회: lawSearch.do + lawService.do
- 행정규칙 포함: admRulSearch.do (시행규칙, 고시 등)
- 자치법규 포함: ordiSearch.do (지역별 조례 차이)

**시장 규모 추정**:
- 연간 신규 사업자등록 약 120만 건
- 기존 자영업자 약 690만명
- TAM: 810만명 x 월 9,900원 = 연 9,600억원 (비현실적 상한)
- SAM: 규제 정보 적극 탐색 10% = 연 960억원
- SOM: 초기 0.01% = 연 960만원

**경쟁사 대비 차별점**:
- 정부24 인허가 안내: 파편화되어 있고 법령 원문 연결 약함
- 창업넷: 개괄 정보만, 법조문 수준 상세 없음
- **차별점**: "업종 -> 법령 조문 -> 체크리스트 -> 변경 알림" 엔드투엔드 경험

### 아이디어 4: 판례 요약 뉴스레터 (CaseBrief)

**한 줄 설명**: 매주 주요 판례를 AI가 요약하여 텔레그램/이메일로 배달하는 구독 서비스

**타겟 사용자**:
- 로스쿨 학생/사법시험 준비생
- 현직 변호사/판사/검사
- 법학 교수/연구자
- 기업 법무팀

**핵심 기능 3가지**:
1. **주간 판례 다이제스트**: 그 주 등록된 주요 판례를 AI가 "쟁점 -> 판단 -> 의미" 구조로 요약
2. **분야별 큐레이션**: 민사/형사/행정/세무/헌법 등 관심 분야만 선택 구독
3. **판례 트렌드**: 월간 법원의 판단 경향 분석 (예: "최근 3개월 개인정보보호법 관련 판례 급증")

**수익 모델**:
- 무료: 주간 TOP 5 판례 요약 (텔레그램)
- 프로: 월 9,900원 - 전체 판례 + 분야별 필터 + 검색 아카이브
- 프리미엄: 월 29,000원 - 판례 분석 리포트 + API 접근

**기술적 구현 난이도**: 중
- 판례 검색: precSearch.do로 최신 판례 수집 (주 1회 batch)
- 판례 본문: precService.do로 전문 조회
- AI 요약: 자체 LLM으로 구조화된 요약 생성
- 배달: 기존 텔레그램 봇 인프라 활용

**시장 규모 추정**:
- 로스쿨 재학생 약 8,000명, 법조인 약 55,000명
- TAM: 63,000명 x 월 9,900원 = 연 75억원
- SAM: 디지털 콘텐츠 구독 의향 30% = 연 22.5억원
- SOM: 초기 2% = 연 4,500만원

**경쟁사 대비 차별점**:
- 빅케이스: 검색 도구 (월 44,000원~), 능동적으로 찾아야 함
- 로앤비: 검색 DB (기관 구독 중심), 뉴스레터 형태 아님
- 케이스노트: 판례 정리 노트 공유 (사용자 생성 콘텐츠)
- **차별점**: "찾아가는" 판례 브리핑 (push), AI 요약 + 트렌드 분석, 저렴한 구독료

### 아이디어 5: 법률 비교 도구 (LawDiff)

**한 줄 설명**: 두 시점의 법령을 비교하여 무엇이 바뀌었는지 한눈에 보여주는 diff 도구

**타겟 사용자**:
- 법률 전문가 (변호사, 법무사, 세무사)
- 국회의원 보좌관/입법조사관
- 기업 법무팀
- 언론사 법조 담당 기자

**핵심 기능 3가지**:
1. **조문 단위 diff**: 개정 전후 조문을 GitHub의 diff 뷰처럼 삭제/추가/변경을 색상으로 표시
2. **개정 히스토리 타임라인**: 특정 법령의 전체 개정 이력을 타임라인으로 시각화
3. **AI 개정 요약**: 개정 내용을 "무엇이, 왜, 어떻게 바뀌었는지" AI가 분석

**수익 모델**:
- 무료: 월 5건 비교
- 프로: 월 14,900원 - 무제한 비교 + AI 분석 + 히스토리
- API: 건당 500원 - 기업/서비스 통합용

**기술적 구현 난이도**: 중
- 법령 연혁: lawHistSearch.do로 개정 이력 조회
- 조문 비교: lawService.do로 각 시점의 법령 본문 조회 + Python difflib으로 diff 생성
- 시각화: GitHub Pages에 정적 diff 뷰어 구현 (JavaScript)
- API 호출 최적화: 한 번 조회한 법령 버전은 영구 캐시 (법령 과거 버전은 변하지 않음)

**시장 규모 추정**:
- 법률 전문가 약 50,000명 + 관련 직종 약 30,000명
- TAM: 80,000명 x 월 14,900원 = 연 143억원
- SAM: 정기 활용 20% = 연 28.6억원
- SOM: 초기 0.5% = 연 1,430만원

**경쟁사 대비 차별점**:
- 국가법령정보센터: 신구조문대비표 PDF만 제공 (불편함)
- 법제처 법제업무지원시스템: 공무원 전용
- **차별점**: 웹 기반 인터랙티브 diff + AI 분석 + 무료/저가

### 아이디어 6: 자치법규 비교기 (LocalLaw)

**한 줄 설명**: 같은 주제에 대해 지역마다 다른 조례를 비교하여 "어디가 유리한지" 알려주는 서비스

**타겟 사용자**:
- 프랜차이즈/다점포 사업자 (지역별 규제 차이 확인)
- 지방의회 의원/보좌관
- 지방자치단체 공무원
- 부동산 개발/투자자

**핵심 기능 3가지**:
1. **지역간 조례 비교**: 같은 주제(예: 간판 규제, 주차장 설치 기준)에 대해 시/도/군별 조례를 나란히 비교
2. **규제 강도 지도**: 전국 지도 위에 특정 규제의 강도를 색상으로 시각화 (열지도)

## TOP 1: 법령 변경 알리미 (LawWatch) - 상세 기획

### 왜 1위인가

- **인프라 완벽 호환**: 현재 law.py의 search_law() + 캐시 DB를 그대로 사용. 텔레그램 봇 인프라도 이미 있음
- **API 호출 최적화**: 워치리스트 법령만 조회 -> 캐시에 이전 버전 저장 -> 변경 시에만 상세 호출. 일 50회 이하로 운영 가능
- **govbrief.kr 시너지**: "정부 보도자료 브리핑 + 관련 법령 변경 알림" = 정책 모니터링 원스톱
- **낮은 구현 난이도**: 2주 내 MVP 가능

### 서비스 아키텍처

```
[사용자] -- 텔레그램 봇 --> [워치리스트 등록]
                                |
                                v
[GitHub Actions cron 06:00] --> [law_watcher.py]
    |                               |
    |   1. 워치리스트 법령 조회      |
    |   2. 캐시와 비교 (diff)       |
    |   3. 변경 감지 시 알림         |
    |                               |
    v                               v
[법제처 API] <-- 캐시 미스 --> [SQLite law_watch.db]
                                |
                                v
                    [텔레그램 알림] + [웹 대시보드]
```

### DB 스키마 설계

```sql
-- 사용자 워치리스트
CREATE TABLE watchlist (
    user_id TEXT,           -- 텔레그램 user_id
    law_name TEXT,          -- 법령명
    keywords TEXT,          -- 추가 키워드 (JSON array)
    created_at TEXT,
    PRIMARY KEY(user_id, law_name)
);

-- 법령 스냅샷 (변경 감지용)
CREATE TABLE law_snapshot (
    law_name TEXT PRIMARY KEY,
    law_mst TEXT,
    enforcement_date TEXT,
    promulgation_date TEXT,
    articles_hash TEXT,     -- 조문 전체의 MD5 해시
    full_text TEXT,         -- 조문 전문 (diff용)
    snapshot_at TEXT
);

-- 변경 이력
CREATE TABLE law_changes (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    law_name TEXT,
    change_type TEXT,       -- 'amendment', 'new', 'abolish'
    old_enforcement TEXT,
    new_enforcement TEXT,
    diff_summary TEXT,      -- AI 생성 변경 요약
    detected_at TEXT
);

-- 알림 발송 기록
CREATE TABLE notifications (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id TEXT,
    law_name TEXT,
    change_id INTEGER,
    sent_at TEXT,
    channel TEXT,           -- 'telegram', 'email'
    FOREIGN KEY(change_id) REFERENCES law_changes(id)
);
```

### 2. API 호출 한도 대응 전략

```
일 100회 한도를 전제로 한 아키텍처 원칙:

1. 캐시 우선 (Cache-First)
   - 법령 텍스트는 30일 캐시 (현재 구현 완료)
   - 법령 과거 버전은 영구 캐시 (신규 추가)
   - 판례 본문은 영구 캐시 (판결 후 변경 없음)

2. 배치 처리 (Batch Processing)
   - 실시간 API 호출 금지 -> 모두 cron batch
   - 일 2회 (06:00, 18:00) 배치로 워치리스트 확인
   - 주 1회 판례 수집

3. 점진적 프리캐시 (Progressive Pre-cache)
   - 인기 법령 100개를 미리 캐시 (일 20개씩 5일)
   - 사용자 요청이 많은 법령부터 캐시 확대
   - 캐시 DB 크기: 법령 1000개 x 평균 50KB = 약 50MB (SQLite 충분)

4. 스마트 변경 감지
   - 전문 조회 대신 "시행일자" 비교만으로 변경 감지
   - 시행일자 변경 시에만 상세 조회 (API 호출 90% 절감)
   - lawSearch.do 1회로 변경 감지 -> lawService.do는 변경 시에만

일 100회 예산 배분:
- 법령변경알리미: 30회 (워치리스트 변경 감지)
- 판례뉴스레터: 10회 (주 1회 batch, 일 평균)
- 법률비교도구: 10회 (캐시 미스 분)
- 예비: 50회 (신규 법령 프리캐시, 긴급 조회)
```

### 3. govbrief.kr 시너지 전략

```
현재 govbrief.kr 구조:
  ├── / (메인: 오늘의 브리핑)
  ├── /weekly (주간 분석)
  └── /finance (금융 특화)

확장 후:
  ├── / (메인: 오늘의 브리핑)
  ├── /weekly (주간 분석)
  ├── /finance (금융 특화)
  ├── /lawwatch (법령 변경 알리미) ← NEW
  ├── /cases (판례 요약) ← NEW
  └── /diff (법률 비교 도구) ← NEW
```

시너지 포인트:
- **크로스 링크**: 보도자료에서 법령 변경 언급 시 -> /lawwatch 해당 법령으로 링크
- **통합 브랜딩**: "govbrief.kr = 정책 + 법률 원스톱 플랫폼"
- **사용자 공유**: 보도자료 독자 -> 법령 알리미 자연 전환
- **SEO 시너지**: "OO법 개정" 검색 -> /diff 페이지 유입 -> 메인 서비스 발견
- **수익화 번들**: 보도자료 프리미엄 + 법령 알리미 프로를 묶어 할인 패키지

## 결론

법제처 Open API만으로 독립 서비스를 만들 때 가장 현실적이고 즉시 실행 가능한 조합은 다음과 같다:

**1순위 - 법령 변경 알리미 (LawWatch)**: 구현 난이도 가장 낮고, 기존 인프라 100% 활용, API 효율 최고. govbrief.kr의 "정책 알림" 정체성과 완벽히 일치.

**2순위 - 판례 요약 뉴스레터 (CaseBrief)**: govbrief.kr의 "AI 요약 -> push 전달" 패턴을 판례 영역으로 확장. 콘텐츠 자산 축적 효과.

**3순위 - 법률 비교 도구 (LawDiff)**: SEO 유입 채널 + 영구 캐시로 API 부담 거의 없음. 한 번 만들면 유지비 거의 0원.

세 서비스를 순차적으로 구축하면 4개월 내에 "정책 + 법률 인텔리전스 플랫폼"으로 포지셔닝 전환이 가능하며, 이는 기존 상업화 PRD의 "금융 규제 모니터링 SaaS" 전략과 자연스럽게 연결된다.
