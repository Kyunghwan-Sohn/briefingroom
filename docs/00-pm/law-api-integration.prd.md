**Author**: PM Agent Team

# Executive Summary

govbrief.kr은 51개 정부 부처 보도자료를 매일 AI 요약하는 서비스다. 보도자료의 상당수는 법령 제/개정, 시행, 해석과 직접 관련되지만, 현재 독자는 해당 법령의 원문이나 맥락을 확인하려면 국가법령정보센터에 별도로 접속해 직접 검색해야 한다.

**핵심 제안**: 법제처 Open API를 연동하여 보도자료에 언급된 법령을 자동 감지하고, 관련 조문/개정이력/해석례를 보도자료 요약과 함께 제공한다. 이를 통해 "정책 발표 -> 법령 근거 확인"의 원스톱 경험을 구현한다.

**API 제약 전제**: 일일 100회 호출 제한 (보수적 설계). 캐싱 + 선별적 호출로 일 150건 보도자료 중 법령 관련 20~30건만 API 호출하되, 캐시 히트율 70% 이상을 목표로 실 호출 10~30회/일로 운영한다.

**기대 효과**:
- 사용자: 보도자료 하나에서 정책 요약 + 법령 근거 + 개정 맥락까지 한 번에 파악
- 서비스: 단순 뉴스 요약에서 "정책 인텔리전스 플랫폼"으로 포지셔닝 전환
- 사업: 상업화 PRD의 금융 규제 모니터링 SaaS와 직접 시너지 (법령 연동 = 유료 기능의 핵심 차별화)

## 2.3 SWOT 분석

### Strengths (강점)
- 51개 부처 보도자료 일일 전수 크롤링 파이프라인 이미 운영 중
- LLM 기반 요약 시스템 안정화 (qwen3.5:35b, 자체 호스팅)
- 법제처 API OC 발급 완료, korean-law-mcp 64개 도구 즉시 사용 가능
- 정적 사이트 + 텔레그램 + RSS 다채널 배포 인프라
- 1인 개발자의 빠른 의사결정과 실행력

### Weaknesses (약점)
- API 일일 호출 제한 (보수적 100회 가정)
- 법률 도메인 전문성 부족 (법령 추출 정확도 검증 필요)
- 1인 운영으로 유지보수/고객지원 한계
- 법령 캐시 DB 구축/운영 부담 추가

### Opportunities (기회)
- 보도자료 + 법령 자동 연결 서비스는 국내에 존재하지 않음
- 금융 규제 모니터링 SaaS (상업화 PRD)의 핵심 차별화 기능으로 활용
- 법제처 API가 무료이므로 한계 비용이 낮음
- AI 기본법 시행 등으로 법령 모니터링 수요 증가 추세
- 공공데이터포털 운영 계정 신청 시 호출 한도 대폭 증가 가능

### Threats (위협)
- CODIT이 보도자료 연동 기능을 추가할 수 있음 (VC 투자 기반)
- 법제처 API 정책 변경 (유료화, 한도 축소) 리스크
- 법령 추출 오류 시 잘못된 정보 제공에 따른 신뢰 하락
- 로앤컴퍼니(빅케이스)가 정책 모니터링 영역으로 확장 가능

### SO 전략 (강점 x 기회)
- 기존 파이프라인에 법령 연동을 "삽입"하여 최소 비용으로 차별화
- 금융 규제 모니터링 SaaS MVP에 법령 연동을 핵심 기능으로 포함

### WT 전략 (약점 x 위협)
- 캐시 DB로 API 의존도 최소화, 법제처 API 정책 변경 대비
- 법령 추출 결과에 "AI 자동 연결" 면책 고지 표시
- 운영 계정 신청으로 호출 한도 확보, 장기적으로 직접 크롤링 백업 구축

## 2.4 기존 상업화 PRD와의 시너지

법령 연동은 상업화 PRD(금융 규제 모니터링 SaaS)와 직접적 시너지를 형성한다:

| 상업화 PRD 기능 | 법령 연동이 제공하는 가치 |
|----------------|----------------------|
| 금융 규제 알림 SaaS | 보도자료 + 관련 법령 조문을 함께 알림 -> 알림의 가치 극대화 |
| 규제 변화 타임라인 | 법령 개정이력 API로 타임라인 자동 구성 |
| 컴플라이언스 체크리스트 | 해당 법령의 조문을 기반으로 체크리스트 자동 생성 |
| 정책 변화 영향 분석 리포트 | 법령 개정 전후 비교 + 관련 판례/해석례 포함 |

**결론**: 법령 연동은 단독 기능이 아니라, govbrief.kr의 모든 서비스를 한 단계 업그레이드하는 **플랫폼 인프라**다.

## 5.1 시스템 아키텍처 개요

```
기존 파이프라인:
  크롤링 -> 파일추출 -> [LLM 요약] -> DB저장 -> WP -> 정적사이트 -> 텔레그램

법령 연동 삽입:
  크롤링 -> 파일추출 -> [LLM 요약 + 법령 추출] -> [법령 API 조회] -> DB저장(확장) -> ...

                                    |
                              +-----v------+
                              | 법령 캐시 DB |
                              |  (SQLite)  |
                              +-----^------+
                                    |
                              +-----v------+
                              | 법제처 API  |
                              | (캐시 미스) |
                              +------------+
```

## 5.2 파이프라인 삽입 시점

기존 `app.py`의 `_process_and_enrich_items()` 함수 내, **LLM 요약 완료 직후, DB 저장 직전**에 법령 연동 단계를 삽입한다.

```python
# 기존 흐름 (app.py 246~249행)
for item in all_items:
    item["summary"] = summarize(item)       # LLM 요약
    time.sleep(0.5)

# === 법령 연동 삽입 ===
from briefingroom.law import enrich_with_law
for item in all_items:
    enrich_with_law(item)                   # 법령 태깅 + API 조회
    time.sleep(0.3)

bulk_upsert(all_items)                      # DB 저장 (법령 정보 포함)
```

**삽입 시점 선택 근거**:
1. LLM 요약 결과에서 법령명을 추출하므로 요약 이후여야 함
2. DB 저장 전에 법령 정보를 item에 추가해야 DB에 함께 저장됨
3. WP 포스팅/텔레그램 발송 전에 법령 정보가 있어야 표시 가능

## 5.4 API 호출 최적화 전략

### 전략 1: 3단계 캐시 계층

```
요청 -> [1] 메모리 캐시 (dict, 당일 세션)
     -> [2] SQLite 캐시 (law_cache, 30일 유효)
     -> [3] 법제처 API (캐시 미스 시에만)
```

- **메모리 캐시**: 같은 실행 세션 내 중복 법령명 제거 (같은 날 여러 보도자료가 같은 법령 언급)
- **SQLite 캐시**: 30일 이내 조회한 법령은 재호출하지 않음 (법령은 자주 바뀌지 않음)
- **API 호출**: 캐시 미스 시에만 실행, 일일 카운터로 100회 제한 감시

### 전략 2: 선별적 호출 (호출 예산 관리)

```python
def should_call_api(item: dict) -> bool:
    """API 호출 여부 결정"""
    # 1. 영향도 "상" 보도자료는 무조건 호출
    if item.get("impact") == "상":
        return True
    # 2. 영향도 "중"이면서 법령 제/개정 키워드 포함 시 호출
    if item.get("impact") == "중":
        keywords = ["개정", "제정", "시행", "공포", "법률", "시행령"]
        if any(kw in item.get("title", "") for kw in keywords):
            return True
    # 3. 영향도 "하"는 법령 링크만 (캐시에 있으면) 제공, API 호출 안 함
    return False
```

### 전략 3: 배치 처리 + 일일 카운터

```python
class LawAPIClient:
    MAX_DAILY_CALLS = 100
    _daily_count = 0
    _daily_date = None

    def call_api(self, law_name: str) -> dict:
        today = date.today()
        if self._daily_date != today:
            self._daily_count = 0
            self._daily_date = today

        if self._daily_count >= self.MAX_DAILY_CALLS:
            print(f"  [법령 API] 일일 한도 도달 ({self.MAX_DAILY_CALLS}회)")
            return None

        # 캐시 확인
        cached = self._check_cache(law_name)
        if cached:
            return cached

        # API 호출
        self._daily_count += 1
        result = self._fetch_from_api(law_name)
        self._save_to_cache(law_name, result)
        return result
```

### 전략 4: 법령명 정규화

LLM이 추출한 법령명은 약칭/풀네임/오타가 혼재할 수 있다. 정규화 단계를 거친다:

```python
LAW_NAME_ALIASES = {
    "자본시장법": "자본시장과 금융투자업에 관한 법률",
    "전금법": "전자금융거래법",
    "개인정보보호법": "개인정보 보호법",
    "산안법": "산업안전보건법",
    # ... 주요 약칭 50개 사전
}

def normalize_law_name(raw_name: str) -> str:
    name = raw_name.strip().replace(" ", "")
    return LAW_NAME_ALIASES.get(name, raw_name.strip())
```

## 5.5 korean-law-mcp vs 직접 API 호출 선택

| 기준 | korean-law-mcp | 직접 API 호출 |
|-----|---------------|-------------|
| 설치 복잡도 | npm 패키지 + MCP 프로토콜 | requests 라이브러리 (이미 의존성) |
| 기능 범위 | 64개 도구 (과잉) | 필요한 3~4개 엔드포인트만 |
| 캐싱 | 내장 (1h/24h) | 직접 구현 (30일 SQLite) |
| 파이프라인 통합 | subprocess/MCP 호출 (복잡) | Python 함수 호출 (자연스러움) |
| 의존성 | Node.js 런타임 필요 | 추가 의존성 없음 |
| 유지보수 | 외부 패키지 업데이트 의존 | 자체 코드 완전 제어 |

**결론: 직접 API 호출 선택**

이유:
1. 기존 파이프라인이 Python (requests 기반)이므로 직접 호출이 자연스러움
2. 필요한 API는 3~4개뿐 (법령 검색, 법령 조문, 개정이력, 판례 검색)
3. 캐시를 SQLite로 직접 관리하면 30일 캐시 등 세밀한 제어 가능
4. Node.js 의존성 추가 없이 운영 가능
5. korean-law-mcp의 64개 도구 중 실제 사용은 5% 미만

단, korean-law-mcp의 법령명 정규화 로직과 API 엔드포인트 매핑은 참고한다.

# Part 8: LLM 프롬프트 설계

## 8.1 확장된 SYSTEM_PROMPT

기존 요약 프롬프트에 법령 추출 필드를 추가한다:

```
당신은 대한민국 정부 정책 전문 분석가입니다.
보도자료 원문을 받으면 반드시 아래 형식으로만 답하세요. 다른 말은 하지 마세요.

요약: (3문장. 핵심 정책 내용 / 주요 수치/시행 시점 / 대상/효과 순서로)
키워드: 키워드1, 키워드2, 키워드3, 키워드4, 키워드5
영향도: 상 또는 중 또는 하
법령: 법령명1, 법령명2 (보도자료에서 직접 언급되거나 근거가 되는 법령의 정식 명칭. 없으면 "없음")

영향도 판단 기준: (기존과 동일)

법령 추출 기준:
- 보도자료 본문에서 직접 언급된 법률, 시행령, 시행규칙의 정식 명칭을 추출
- 약칭이 사용된 경우 정식 명칭으로 변환 (예: "자본시장법" -> "자본시장과 금융투자업에 관한 법률")
- 제/개정, 시행, 공포 등 법령 관련 동작이 언급된 경우 우선 추출
- 단순 참조(예: "OO법에 따라")도 포함
- 최대 3개까지만 추출 (가장 관련도 높은 순)
- 법령과 관련 없는 보도자료는 "없음"
```

## 8.2 LLM 출력 파싱

```python
def parse_law_names(summary_text: str) -> list[str]:
    """LLM 요약 결과에서 법령명 목록 추출"""
    for line in summary_text.split("\n"):
        if line.startswith("법령:"):
            raw = line.replace("법령:", "").strip()
            if raw == "없음" or not raw:
                return []
            return [name.strip() for name in raw.split(",") if name.strip()]
    return []
```
